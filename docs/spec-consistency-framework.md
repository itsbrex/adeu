# Specification: Polyglot Consistency Framework

## 1. Executive Summary

**The Problem:** Adeu maintains two parallel redlining engines: Python (`adeu-py`) and TypeScript (`@adeu/core`). Because Adeu operates as a surgical XML patcher, both engines must achieve 100% output parity. Unit tests isolated by language cannot guarantee that given the exact same input DOCX and the exact same `ModifyText` payload, both engines will produce mathematically identical Virtual DOM abstractions and OOXML mutations.

**The Solution:** We are implementing a **Data-Driven Golden Master** testing framework. We will establish a language-agnostic shared corpus of test cases. The Python engine (which is historically battle-tested) will act as the "Judge", generating Ground Truth snapshots. The Node.js engine will act as the "Challenger", dynamically executing the corpus and asserting its outputs against the Python snapshots.

## 2. Architecture: Two-Tiered Parity

### Tier 1: The Shared JSON Corpus (Core Engine Parity)
We will maintain a suite of test scenarios in `shared/cross_platform_tests/`. 

Each test scenario is a directory containing:
1. `input.docx`: The starting document.
2. `test.json`: The specific configuration and edit payload.
3. **Golden Masters** (Generated by Python):
   - `golden_raw.md`: The output of `extract_text_from_stream(clean_view=False)`.
   - `golden_clean.md`: The output of `extract_text_from_stream(clean_view=True)`.
   - `golden_abstract.xml`: The post-edit abstracted XML generated by `xml_debug.py`.

### Tier 2: MCP Dual-Client Harness (RPC Parity)
To ensure the outer API layer (FastMCP vs. `@modelcontextprotocol/sdk`) behaves identically, we will create an End-to-End script (`node/scripts/mcp-parity.ts`). This script will boot both servers via `stdio`, send identical JSON-RPC tool calls (`read_docx`, `process_document_batch`), and assert that the `ToolResult` JSON payloads match exactly.

## 3. The `test.json` Schema

The `test.json` file dictates what the engine should do to the `input.docx`.

```json
{
  "name": "basic_text_modification",
  "description": "Tests replacing a single word in a standard paragraph.",
  "author": "Adeu AI",
  "read_only": false,
  "changes": [
    {
      "type": "modify",
      "target_text": "Vendor",
      "new_text": "Supplier",
      "comment": "Updated terminology"
    }
  ]
}
```
*If `read_only` is true, the engines will only test extraction parity (skipping the `process_batch` and XML abstraction phases).*

## 4. Workflows

### 4.1 Updating the Ground Truth
When core logic changes (e.g., tweaking how `trim_common_context` handles markdown), the Golden Masters must be regenerated.
```bash
# Executed by developers when altering expected behavior
uv run python/scripts/generate_goldens.py
```
This script will loop through `shared/cross_platform_tests/`, read `test.json`, apply the edits using Python's `RedlineEngine`, and overwrite the `golden_*.md` and `golden_abstract.xml` files.

### 4.2 Running the Checks (CI/CD)
The Node test suite will dynamically read the directories inside `shared/cross_platform_tests/`.
```bash
# Executed by CI and developers
npm run test --workspace=@adeu/core
```
The `vitest` runner will:
1. Load `input.docx`.
2. Apply `test.json` using the TS `RedlineEngine`.
3. Assert TS Markdown projection `===` Python's `golden_*.md`.
4. (Optional but recommended) Shell out to `python/scripts/abstract_xml.py` to abstract its resulting TS DOCX, and assert it matches `golden_abstract.xml`.

## 5. Overcoming XML Impedance Mismatch
Direct byte-for-byte DOCX or XML comparisons will fail because `lxml` (Python) and `@xmldom/xmldom` (Node) serialize XML differently (e.g., `<w:b/>` vs `<w:b></w:b>`, attribute ordering, namespace prefixes).

We bypass this using Python's existing `abstract_docx_xml` logic from `xml_debug.py`. It normalizes relationships, strips volatile RSIDs/Timestamps/session IDs, and alphabetically sorts attributes, allowing a mathematically secure diff of the underlying structure.
